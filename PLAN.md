# Plan: piloted-agent dataset + server-side cache instrumentation

## Why this pivot

Current state (after the multi-splice work in `reagent/`):

- The cross-session splice experiments on `nemotron-swe/pandas-dev` measured
  matches that all turned out to be inside the first turn (system + first
  user message), i.e. the prefix-cache regime, not the body-splice regime.
- `swe-smith/django` does have legitimate second-turn-onward matches (59 of
  190 ordered pairs after the role + first-turn-skip filters), but the
  body matches are 1-2k token chunks scattered across 11k of trajectory —
  borderline economics for disk cache (~1k-token break-even with NVMe
  load), and the trajectories were generated by GPT-4-shaped agents whose
  output structure is opaque to us.
- We do not control the prompt composition. We can't tell the agent
  framework "emit your sections with structural identity" because we are
  consuming the rendered chat-template output.
- We've been measuring noise on too-short trajectories without knowing it,
  driven by datasets chosen for availability rather than fit.

The right move: **build our own data**, with full control over (a) the
agent framework's structural metadata, (b) the model serving the agent,
and (c) the tasks driving cross-session reuse. This is the same pattern
we used to validate lmcache earlier (`lmcache/prompts/`).

## Stack

```
┌─────────────────────────────────────────────────────────────┐
│ Driver (Claude, acting as the user)                         │
│   Sends realistic SWE tasks into the agent CLI;             │
│   re-uses tasks across sessions to drive cross-session      │
│   matches.                                                  │
├─────────────────────────────────────────────────────────────┤
│ Agent CLI (OpenCode or Hermes code agent)                   │
│   Loads tool definitions, AGENTS.md, skills.                │
│   Issues tool calls (read_file, grep, edit, run, ...) and   │
│   composes the chat history.                                │
│   Talks HTTP (OpenAI-compat) to the server.                 │
├─────────────────────────────────────────────────────────────┤
│ Server: `transformers serve` (forked / patched)             │
│   Receives `chat/completions` requests with structured      │
│   `messages` + `tools`. INTERCEPTS HERE to:                 │
│     - tag turn boundaries (system / user / tool / assistant)│
│     - identify stable structural sections                   │
│       (system prompt vs tools vs AGENTS.md vs skills vs     │
│        per-session injections)                              │
│     - emit a per-request manifest alongside the rendered    │
│       prompt and per-token role/section labels              │
│   This is the only realistic place to capture metadata that │
│   the agent already has but doesn't expose.                 │
├─────────────────────────────────────────────────────────────┤
│ Disk cache (next phase)                                     │
│   Per-section K/V offload keyed by structural identity,     │
│   not by byte-hash. Built on the manifest the server emits. │
└─────────────────────────────────────────────────────────────┘
```

Picking `transformers serve`:
- HF-native, easiest path to extend without going through vLLM internals.
- Already exposes the `chat/completions` shape we need.
- The forwarding loop is a single well-known function we can wrap to
  emit metadata before the K/V is computed.
- Avoids fighting paged-attention managers (we'll get to vLLM later
  once the metadata story is clean).

## Phase 1 — single-session smoke (no caching)

Goal: a piloted agent CLI run end-to-end against `transformers serve`,
emitting (a) the rendered prompt, (b) per-token role labels, (c) a
manifest of structural sections.

Deliverables:

1. Pick agent CLI: try OpenCode first (the user has used it before, and
   `lmcache/opencode.json` is already present). Hermes code agent as
   fallback.
2. Stand up `transformers serve` on a small model that has tool support
   (Qwen2.5-Coder-7B-Instruct or similar — ChatML-native so OpenCode's
   tool-rendering doesn't need work).
3. Drive a single small task ("read README.md and summarize"). Save the
   chat trace.
4. Patch the serve loop to emit:
   - `messages_in.json` — the structured input as received
   - `prompt.txt` — the rendered chat-template output
   - `role_per_token.json` — per-token role labels
   - `section_manifest.json` — see below
5. Verify by hand that the manifest correctly identifies system /
   tools / AGENTS.md / per-session injections.

## Section manifest contract

For each chat completion, the server emits a manifest:

```json
{
  "model_id": "...",
  "agent_build_id": "...",         // hashed from system + tools text
  "project_id": "...",             // path of the repo the agent is in
  "skill_set": ["skill-foo@v3", "..."],
  "sections": [
    {"id": "system",     "role": "system", "tok_range": [0, 740]},
    {"id": "tools",      "role": "system", "tok_range": [740, 1820]},
    {"id": "agents-md",  "role": "system", "tok_range": [1820, 2104]},
    {"id": "skill:foo",  "role": "system", "tok_range": [2104, 2380]},
    {"id": "user-task",  "role": "user",   "tok_range": [2380, 2415]}
  ]
}
```

Section identification heuristics (in order of confidence):

1. **From `tools=` parameter** — if the OpenAI-compat request carries
   tools as a structured list (Llama 3.1+ / Mistral / Qwen ChatML
   templates), the rendered token range of the tool block is computed
   by rendering with-and-without `tools=`.
2. **Repeated-prefix detection** — across multiple requests in the same
   session, the longest stable prefix is the agent build (system +
   tools + AGENTS.md + skills, all of which are static). Across
   different projects on the same agent, the divergence point is the
   AGENTS.md / project-memory boundary.
3. **Heuristic markers** — XML tags like `<system-reminder>`,
   `<context>`, OpenCode-specific delimiters (need to be discovered
   from agent output). These get tagged as "session-varying", so the
   cache layer knows to refuse them.

The first run won't get all of this right. Phase 1's goal is to emit
the *easy* layers (turn boundaries, role labels, top-level system /
tools split) and leave the finer-grained tagging as a follow-up once
we have real traces to mine.

## Phase 2 — multi-session same-project run

Goal: generate enough sessions on the same project to measure both
prefix-cache hits (first turn) and body-splice opportunities
(recurring tool results).

Driver protocol (Claude as user):

- Pick a small but real codebase (the `lmcache-src/` submodule is a
  candidate; or a vendored small Python project).
- Define ~10 tasks against the same codebase, designed so their tool
  calls overlap:
  - "explain the cache eviction logic" (reads core files)
  - "add a metric for cache hit rate" (reads + edits)
  - "find all places that call `evict_lru`" (grep + reads)
  - "write a test for evicting under memory pressure" (reads + new file)
  - ... (mix of read-heavy and edit-heavy, all touching shared files)
- Each task starts a fresh agent session. AGENTS.md is identical
  across all 10 sessions (project-static).
- Capture each session as `(messages, manifest, rendered_prompt)`.

Expected match landscape (what we'd want to confirm):

- **First turn (positions ~0-3k)**: byte-identical across all 10
  sessions → 100% prefix-cache hit.
- **Tool-result chunks** (mid-trajectory, positions 3k-30k): some
  recur because multiple sessions read the same files. These are
  the body-splice candidates.

If this works, we can finally answer "how often do real tool-result
chunks recur and how big are they?" — the question that motivated
multi-splice in the first place.

## Phase 3 — cross-project, cross-agent runs

To stress the structural-identity cache key:

- Same agent, two projects (different AGENTS.md / project state).
- Two agents (OpenCode + Hermes) on the same project (different
  system prompts, same project memory).
- Tool-set perturbations: same agent build with one tool added.

Each crossover tests a different cache-eviction boundary. The point
isn't to maximize hits; it's to confirm the cache key correctly
isolates which sessions can share what.

## Phase 4 — splice-tolerance experiments on the new data

Re-run the existing reagent experiments on this data:

- `measure_multi_splice_b.py` with the role + first-turn filters
  (already shipped).
- `experiment_kv_perturbation.py` for write-time cacheability.
- `analyze_splice_predictors.py` for read-time text similarity.

The goal isn't to add new analyses; it's to redo the existing ones on
data that actually has post-first-turn body matches with realistic
tool-result shape.

## Phase 5 — disk-cache layer (out of scope for this plan, but the
target)

Once we have:
- A serve loop that emits structural section manifests
- Datasets where second-turn-onward chunks recur with measurable size

We can wire a persistent disk cache that:
- Keys K/V slices by structural section id, not byte hash
- Refuses to write per-session sections to disk
- Versions by model weights id

This is the actual product question. Everything in phases 1-4 is
infrastructure to make phase 5 measurable rather than guessed.

## Open decisions

- **Which agent CLI**: OpenCode is the more developed option and we
  have an `opencode.json` already. Hermes code agent as backup.
- **Which model**: needs tool calling, not too big to slow down
  iteration. Qwen2.5-Coder-7B-Instruct is the leading candidate;
  Llama-3.1-8B-Instruct as fallback. Note: we want a model whose chat
  template handles `tools=` structurally, otherwise tool-section
  identification falls back to heuristics.
- **Where the patched serve lives**: forked module under
  `reagent/serve/` (or a new top-level `tserve/` directory) rather
  than in-tree patching of the transformers package. Easier to
  rebase against upstream.
- **Storage layout for traces**: one directory per session with
  `messages.json`, `manifest.json`, `prompt.txt`, `tool_io/*` for
  the raw tool inputs/outputs. Lets us replay or inspect any single
  session without the server.

## Anti-goals

- Not building a competitor to vLLM, lmcache, sglang, or paged
  attention managers. The serve fork is a research vehicle for
  metadata extraction.
- Not training anything. No SFT / RLHF. The agent's behavior is
  whatever the chosen open model produces; we just want realistic
  traces, not high-quality task completion.
- Not solving the cacheability prediction problem in the general
  case (we showed it's intrinsically hard with text-only signals).
  We're solving the *structural identification* problem, which is
  tractable with framework cooperation.

## What "done" looks like for phase 1

A single command produces:

```
runs/2026-04-30-smoke/
  001/
    messages.json        # the OpenAI-compat input
    manifest.json        # section breakdown
    prompt.txt           # rendered chat template
    role_per_token.json  # 0..N labels
    response.txt         # the model's reply
```

And we can spot-check a single session by hand and see that the
manifest correctly tags `[system, tools, agents-md, user-task]` with
plausible token ranges.
